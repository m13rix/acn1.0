# ACN 1.0 - Техническое описание фреймворка

## Обзор

**ACN (Agentic AI Framework)** - это высокопроизводительный фреймворк для создания и управления агентами на основе больших языковых моделей (LLM). Фреймворк предоставляет модульную архитектуру с поддержкой различных провайдеров моделей, гибкой системой инструментов, изолированными средами выполнения и встроенной системой знаний.

## Архитектура

### Основные компоненты

Фреймворк построен на модульной архитектуре с четким разделением ответственности:

1. **Session (Сессия)** - управляет состоянием агента, историей разговора и жизненным циклом песочницы
2. **Executor (Исполнитель)** - оркестрирует основной цикл работы агента
3. **Provider (Провайдер)** - абстракция над различными LLM API
4. **Syntax (Синтаксис)** - определяет формат коммуникации между агентом и фреймворком
5. **Loop (Цикл)** - определяет логику выполнения действий и продолжения разговора
6. **Sandbox (Песочница)** - изолированная среда для выполнения кода
7. **Skills System (Система знаний)** - система управления знаниями на основе векторного поиска

---

## Детальное описание компонентов

### 1. Session (Сессия)

**Расположение:** `src/core/Session.ts`

Класс `Session` является центральным менеджером состояния агента. Каждая сессия:

- **Хранит историю сообщений** - массив сообщений ролей: `system`, `user`, `assistant`, `file`
- **Управляет песочницей** - создает и очищает изолированную директорию для выполнения кода
- **Инициализирует SkillsService** - если у агента настроена таблица знаний (`skillsTable`)
- **Строит системный промпт** - комбинирует базовый промпт агента, документацию синтаксиса, цикла и инструментов

**Ключевые методы:**
- `initialize()` - инициализация песочницы и сервисов
- `addUserMessage()` / `addAssistantMessage()` / `addFileMessage()` - добавление сообщений
- `getAllMessages()` - получение всех сообщений включая системный промпт
- `getProviderConfig()` - генерация конфигурации провайдера из настроек агента

**Особенности:**
- Отслеживание добавленных навыков (`addedSkillIds`) для предотвращения дублирования в контексте
- Автоматическая генерация уникального ID сессии
- Поддержка файловых сообщений для передачи контента (изображения, PDF, видео)

---

### 2. Executor (Исполнитель)

**Расположение:** `src/core/Executor.ts`

Класс `Executor` - это основной оркестратор цикла агента. Он реализует следующий цикл:

1. **Поиск навыков** - автоматический поиск релевантных знаний из базы перед отправкой сообщения
2. **Вызов провайдера** - отправка сообщений в LLM через выбранный провайдер
3. **Обработка ответа** - парсинг ответа с использованием синтаксиса для извлечения действий
4. **Выполнение действий** - выполнение кода в песочнице или CLI команд
5. **Продолжение цикла** - формирование сообщения с наблюдением и продолжение до завершения

**Ключевые особенности:**

- **Стриминг** - поддержка потокового вывода с отдельными событиями для reasoning и текста
- **Автоматический поиск навыков** - перед каждым пользовательским сообщением выполняется поиск релевантных знаний (порог сходства 80%)
- **Обработка ошибок** - graceful handling ошибок выполнения с передачей в контекст агента
- **Колбэки** - расширенная система колбэков для мониторинга и отладки

**Параметры:**
- `maxIterations` - максимальное количество итераций цикла (по умолчанию 10)
- `stream` - включение потокового вывода
- `callbacks` - колбэки для различных событий

---

### 3. Provider (Провайдер)

**Расположение:** `src/providers/`

Провайдеры - это абстракция над различными LLM API. Все провайдеры реализуют универсальный интерфейс `Provider`:

```typescript
interface Provider {
  name: string;
  complete(messages: Message[], config: ProviderConfig): Promise<ProviderResponse>;
  streamEvents?(messages: Message[], config: ProviderConfig): AsyncIterable<ProviderStreamEvent>;
  buildRequest?(messages: Message[], config: ProviderConfig): any;
}
```

**Универсальная конфигурация:**
Все провайдеры используют единый формат конфигурации `ProviderConfig`:
- `model`, `temperature`, `maxTokens`
- `top_p`, `top_k`, `seed` - параметры сэмплирования
- `frequency_penalty`, `presence_penalty`, `repetition_penalty` - штрафы
- `reasoning` - уровень reasoning для моделей с этой функцией
- `stream` - потоковый режим

**Реализованные провайдеры:**
- **GeminiProvider** (`src/providers/gemini.ts`) - Google Gemini API
- **OpenRouterProvider** (`src/providers/openrouter.ts`) - OpenRouter API (доступ к множеству моделей)

**Регистрация провайдеров:**
Провайдеры регистрируются через систему реестра:
```typescript
registerProvider('gemini', () => new GeminiProvider());
```

**Стриминг:**
Провайдеры поддерживают современный стандарт стриминга с событиями:
- `reasoning.delta` / `reasoning.done` - для reasoning контента
- `text.delta` / `text.done` - для текстового контента
- `done` - завершение стрима

---

### 4. Syntax (Синтаксис)

**Расположение:** `src/syntax/`

Синтаксис определяет формат коммуникации между агентом и фреймворком. Каждый синтаксис должен реализовать интерфейс `SyntaxType`:

**Методы извлечения:**
- `getThinking()`, `getAction()`, `getObservation()`, `getCli()`, `getSkills()` - извлечение контента из тегов
- `hasAction()`, `hasCli()` - проверка наличия тегов

**Методы обертки:**
- `wrapThinking()`, `wrapAction()`, `wrapObservation()`, `wrapCli()`, `wrapSkills()` - обертка контента в теги

**Реализованные синтаксисы:**

**XMLTagsSyntax** (`src/syntax/xml-tags.ts`):
Использует XML-подобные теги:
- `<think>...</think>` - для размышлений агента
- `<action>...</action>` - для кода выполнения
- `<obs>...</obs>` - для наблюдений (вставляются системой)
- `<cli>...</cli>` - для CLI команд
- `<skills>...</skills>` - для знаний из базы

**Особенности:**
- Обработка неполных тегов (когда закрывающий тег обрезан stop sequence)
- Поддержка извлечения даже при обрезанных тегах
- Автоматическая генерация документации для системного промпта

---

### 5. Loop (Цикл)

**Расположение:** `src/loops/`

Циклы определяют логику выполнения действий и построения продолжения разговора. Интерфейс `LoopType`:

**Основные методы:**
- `processResponse()` - обработка ответа модели и извлечение действий
- `buildContinuationMessages()` - построение сообщений для продолжения после выполнения действия
- `getDescription()` - документация для системного промпта
- `shouldCommitMessagesAfterAction()` - определяет, коммитить ли сообщения после каждого действия

**Реализованные циклы:**

**AccumulatorLoop** (`src/loops/accumulator.ts`):
- Накопление всего вывода агента в одном сообщении assistant
- После выполнения действия наблюдение добавляется к текущему сообщению
- Продолжение с пользовательским сообщением "Complete the user request"
- Stop sequences: `</action>`, `</cli>`

**MessagePassthroughLoop** (`src/loops/message-passthrough.ts`):
- Каждое действие коммитится как отдельное сообщение
- Наблюдения добавляются как пользовательские сообщения
- Более линейная история разговора

---

### 6. Sandbox (Песочница)

**Расположение:** `src/sandbox/Sandbox.ts`

Класс `Sandbox` создает изолированную среду выполнения для каждого агента:

**Функциональность:**

1. **Создание изолированной директории** - каждый сеанс получает свою директорию (`session-{id}`)
2. **Настройка окружения** - создание `package.json` и `tsconfig.json` для TypeScript выполнения
3. **Импорт инструментов** - автоматическая генерация импортов для всех доступных инструментов
4. **Выполнение кода** - выполнение TypeScript через `tsx`
5. **Выполнение CLI** - выполнение shell команд в контексте песочницы

**Особенности:**

- **Поддержка diff-редактирования** - если код содержит diff формат, применяется редактирование существующего файла
- **Отслеживание выполненных файлов** - для поддержки diff по именам файлов
- **Системный инструмент** - автоматический импорт системного инструмента для управления знаниями
- **Очистка** - автоматическая очистка директории при завершении сессии

**Безопасность:**
- Изоляция на уровне файловой системы (отдельная директория)
- Ограниченный доступ к инструментам (только загруженные инструменты доступны)

---

### 7. Skills System (Система знаний)

**Расположение:** `src/skills_system/`

Система знаний использует векторный поиск для хранения и извлечения знаний агента.

**Архитектура:**

1. **Embeddings** (`embeddings.ts`):
   - Использует Google Gemini Embedding API (`gemini-embedding-001`)
   - Кэширование эмбеддингов для избежания повторных вызовов API
   - Батчовая обработка для эффективности

2. **LanceDB** - векторная база данных:
   - Хранение навыков с эмбеддингами
   - Таблица на агента (`skillsTable` в конфигурации)
   - Схема: `id`, `content`, `examples`, `exampleVectorsJson`, `scoreThreshold`, `updatedAt`

3. **Retriever** (`retriever.ts`):
   - Пример-ориентированный семантический поиск
   - Сравнение запроса пользователя с примерами каждого навыка
   - Возвращает топ результатов с оценками сходства (cosine similarity)

4. **SkillsService** (`SkillsService.ts`):
   - Центральный сервис для управления навыками
   - `addEntry()` - добавление навыка с примерами
   - `search()` - поиск релевантных навыков
   - Автоматическая инициализация таблицы при первом использовании

**Работа с навыками:**

- **Добавление:** Агент добавляет навык через системный инструмент `skills.add(content, examples, threshold)`
- **Автоматический поиск:** Перед каждым пользовательским сообщением Executor автоматически ищет релевантные навыки
- **Порог сходства:** По умолчанию 80% для автоматического извлечения
- **Предотвращение дублей:** Система отслеживает уже добавленные навыки в контекст

## Конфигурация агента

Агенты конфигурируются через YAML файлы (`agent.yaml`):

```yaml
name: test-agent
description: A test agent

model: models/gemini-3-flash-preview
provider: gemini
temperature: 1
maxTokens: 4096

systemPrompt: system.md  # Файл с системным промптом

tools:
  - search  # Список инструментов

loop: message-passthrough  # Тип цикла
syntax: xml-tags           # Тип синтаксиса

skillsTable: test  # Опционально: имя таблицы знаний
```

**Компоненты:**
- `name`, `description` - идентификация агента
- `model`, `provider` - модель и провайдер
- Параметры модели (`temperature`, `maxTokens`, и т.д.)
- `systemPrompt` - файл с системным промптом
- `tools` - список инструментов
- `loop`, `syntax` - выбор цикла и синтаксиса
- `skillsTable` - опциональная таблица знаний

## Система инструментов

**Расположение:** `tools/`

Инструменты - это расширяемые функции, доступные агенту. Каждый инструмент:

1. **Конфигурация** (`tool.yaml`):
   ```yaml
   name: skills
   description: |
     Built-in skills tool for knowledge base management
   module: index.ts
   ```

2. **Реализация** (`index.ts`):
   - TypeScript модуль, экспортирующий функции
   - Функции автоматически документируются из JSDoc комментариев
   - Доступны в песочнице через импорты

**Встроенные инструменты:**

- **skills** (`tools/skills/`) - управление знаниями:
  - `skills.add()` - добавление навыка
  - `skills.search()` - поиск в базе знаний
  - `skills.viewFiles()` - чтение файлов из песочницы

- **search** (`tools/search/`) - веб-поиск через Exa API

- **math** (`tools/math/`) - математические функции

**Загрузка инструментов:**

`ToolLoader` (`src/loaders/ToolLoader.ts`) рекурсивно сканирует директорию `tools/` и загружает конфигурации. Документация извлекается из JSDoc комментариев и добавляется в системный промпт.

---

## Загрузка агентов

**Расположение:** `src/loaders/AgentLoader.ts`

`AgentLoader` рекурсивно сканирует директорию `agents/`:

- Ищет `agent.yaml` или `agent.yml` в каждой директории
- Загружает системный промпт из указанного файла
- Кэширует загруженных агентов
- Поддерживает вложенные директории

**API:**
- `loadAll()` - загрузка всех агентов
- `loadByName(name)` - загрузка конкретного агента
- `getAvailableAgents()` - список доступных агентов

---

## Система реестров

Фреймворк использует систему реестров для динамической загрузки компонентов:

- **Provider Registry** - реестр провайдеров
- **Syntax Registry** - реестр синтаксисов
- **Loop Registry** - реестр циклов

Компоненты регистрируются при импорте модулей (`src/providers/index.ts`, и т.д.) и доступны через функции `getProvider()`, `getSyntax()`, `getLoop()`.

## Система сборки системного промпта

**Расположение:** `src/core/PromptBuilder.ts`

`PromptBuilder` комбинирует различные части системного промпта:

1. Базовый системный промпт из файла агента

## Особенности реализации

### Обработка неполных тегов

Система синтаксиса обрабатывает случаи, когда закрывающие теги обрезаны stop sequence. Используются два паттерна:
1. Полный тег: `<tag>content</tag>`
2. Неполный тег: `<tag>content` (без закрывающего тега)

### Дифференциальное редактирование

Песочница поддерживает diff-формат для редактирования существующих файлов:
```typescript
// Формат: @@filename
// Старый код
// - строка для удаления
// + строка для добавления
```

### Потоковый вывод

Поддержка современного стандарта стриминга с раздельными событиями для reasoning и текста, что позволяет моделям показывать процесс размышления отдельно от финального ответа.

### Кэширование эмбеддингов

Система знаний кэширует эмбеддинги в памяти для избежания повторных API вызовов при одинаковых текстах.

### Изоляция сессий

Каждая сессия получает свою изолированную директорию песочницы, что позволяет параллельную работу нескольких агентов.

## Расширяемость

Фреймворк спроектирован для легкого расширения:

1. **Добавление провайдера:** Реализовать `BaseProvider`, зарегистрировать через `registerProvider()`
2. **Добавление синтаксиса:** Реализовать `BaseSyntax`, зарегистрировать через `registerSyntax()`
3. **Добавление цикла:** Реализовать `BaseLoop`, зарегистрировать через `registerLoop()`
4. **Добавление инструмента:** Создать директорию в `tools/` с `tool.yaml` и `index.ts`

Все компоненты используют систему реестров для динамической загрузки, что позволяет добавлять новые компоненты без изменения ядра фреймворка.
