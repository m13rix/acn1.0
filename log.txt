1. Build a real “Agent Workbench” UI

You already have fragments: memory-viz, skills-viewer, imported-tools-studio, demo HTML UIs, adaptive-step-context debug, Telegram UI bridge callbacks. I would unify them into one serious web UI.

Core panels:

• Conversation / reasoning / tool stream timeline
  • User messages, assistant text, reasoning summaries, tool calls, observations.
  • Expand full truncated observations from data/tool-output/....
• Live workspace panel
  • File tree, current diffs, changed files, generated artifacts.
  • “Review patch” and “revert hunk/file” controls.
• Terminal/session panel
  • Named persistent terminals, server logs, running jobs.
• Subagent panel
  • Background jobs from agents.start, statuses, traces, changed files.
• Memory/context panel
  • Memory hints used, search queries, scores, adaptive step heat map.
• Cost/latency panel
  • Per step token usage from CostLedger, model, provider, cache hits, duration.
• Human interrupt
  • Pause/continue, approve risky command, nudge agent, inject additional context.

Why this matters: coding agents fail less when the human can see exactly what they are doing and intervene cheaply. A high-quality workbench would also make self-improvement much easier because every run becomes inspectable.

2. Add a first-class trace/event ledger

Right now events are distributed through callbacks, cost logs, adaptive context, terminal output, and session snapshots. I’d introduce a canonical append-only trace format:

ts
{
  id,
  sessionId,
  parentSessionId?,
  agentName,
  timestamp,
  type:
    | "model.request"
    | "model.response"
    | "tool.call"
    | "tool.result"
    | "file.read"
    | "file.edit"
    | "terminal.run"
    | "test.result"
    | "memory.search"
    | "subagent.started"
    | "subagent.finished"
    | "checkpoint",
  payload,
  summary,
  artifactRefs,
  cost?,
  durationMs?
}

Then every UI, replay tool, eval harness, and self-improver can consume the same timeline.

This would improve coding performance because failures become diagnosable: bad context, bad model, missed file, bad edit, wrong verification, overlong observation, etc.

3. Create a coding-agent evaluation suite

The biggest missing piece for “standards or better” is objective measurement.

Create a benchmark folder like:

txt
evals/coding/
  fixtures/
  tasks/
    edit-small-bug.yaml
    add-feature-with-tests.yaml
    refactor-preserve-api.yaml
    investigate-no-change.yaml
    multi-file-type-error.yaml
  runner.ts
  scorer.ts

Each task should define:

• initial repo fixture
• user request
• expected changed files or forbidden changed files
• required tests
• semantic assertions
• max tool calls / max cost optional
• scoring rubric

Metrics:

• task success
• tests passed
• compile passed
• number of unnecessary files touched
• number of failed tool calls
• number of user clarifications
• wall time
• cost
• whether final answer was truthful about verification

This would let you compare prompts, models, tool changes, adaptive-context settings, and memory behavior. Without this, “improvement” stays vibes-based.

4. Add structured “patch plans” and verification contracts

For coding tasks, the harness could encourage/require a small internal contract before edits:

ts
{
  intent: "fix failing test X",
  filesToInspect: [],
  filesToModifyExpected: [],
  riskyAreas: [],
  verificationCommands: [],
  rollbackPlan: []
}

Then after the run:

ts
{
  changedFiles: [],
  verificationRun: [],
  verificationStatus: "passed" | "failed" | "not-run",
  userChangeProtection: "clean" | "unrelated-changes-detected",
  summary: ""
}

This does not need to be exposed as verbose final output every time, but the harness should log it. The agent can still be autonomous, but with better self-discipline.

This would reduce common coding-agent failures:

• editing before understanding
• forgetting verification
• claiming tests passed without running them
• accidentally touching unrelated user changes
• making broad rewrites instead of surgical edits

5. Upgrade code from outline-only to semantic code intelligence

The current code.outline() is useful, but coding performance would jump if the harness had language-server-like tools:

Suggested APIs:

ts
await code.symbols(query)
await code.references(file, symbolOrPosition)
await code.definition(file, line, column)
await code.callGraph(file, symbol)
await code.diagnostics({ changedOnly?: true })
await code.dependencyMap(file)
await code.findTestsFor(file)
await code.publicApi(file)

Even a TypeScript-only implementation using the compiler API would be valuable.

Why this matters: agents often waste context reading whole files. Semantic navigation would let me answer questions like “where is this method used?” or “which tests cover this function?” precisely.

6. Improve observation compaction into structured summaries

ToolExecutionEngine.compactObservationForModel() currently keeps head/tail and stores full output. That’s good, but for coding it could be much smarter.

For terminal/test output, generate structured summaries:

ts
{
  command,
  success,
  exitCode,
  durationMs,
  errorCount,
  failedTests: [],
  typeErrors: [],
  firstRelevantStackTrace,
  fullOutputRef
}

Then present the model:

txt
Command failed.
Relevant errors:
1. src/foo.ts:42 TS2345 ...
2. tests/bar.test.ts:17 expected ...
Full output: data/tool-output/...

This would prevent important compiler/test errors from being lost in huge logs and reduce repeated command runs.
7. Make file edits more agent-safe and reviewable

files.edit is good because it forces exact replacements. I’d add higher-level safe editing support:

ts
await files.patch({
  file,
  intent,
  edits: [{ old, new }],
  requireUniqueOld: true,
  verifySyntax: true
})

And add automatic post-edit metadata:

• changed line ranges
• whether replacement was unique
• nearby syntax diagnostics
• whether EOL style was preserved
• whether file was user-modified since last read

Also useful:

ts
await files.diff({ since: "turn-start" })
await files.revert({ file, hunkId })
await files.changedFiles({ includeIgnored: false })

Coding agents are only as good as their edit primitive. A better patch/diff layer would directly improve reliability.

8. Add workspace snapshot and rollback support

Before non-trivial edits, the harness could create a lightweight snapshot:

• git diff baseline
• file hashes for touched files
• optional temp copy of files

Then expose:

ts
await workspace.snapshot("before-fix")
await workspace.diff("before-fix")
await workspace.rollback("before-fix", { files })

This is especially important because the current workspace can contain many unrelated dirty files. The agent needs a robust way to avoid and recover from accidental interference.

9. Better user-change protection

The prompt already says “protect unrelated user changes,” but the harness can enforce it.

Possible mechanism:

• At turn start, record git status --porcelain and hashes of dirty files.
• If agent edits a pre-dirty file, require it to have read that file first.
• Before final response, warn if changed files include files not touched by the task plan.
• Provide a clean summary:

txt
Changed by this turn:
- src/x.ts
Pre-existing dirty files not touched:
- agents/core/prompts/system.md
...

This would be a big professional-grade coding-agent improvement.

10. Add dynamic model/reasoning routing by task phase

The Telos-Code agent currently uses gpt-5.5 with low reasoning and has model switching config. I’d make routing phase-aware:

Phases:

• repo exploration
• planning
• surgical edit
• debugging failed tests
• final summarization
• large refactor
• subagent delegation

For each phase, choose:

• model
• reasoning effort
• max output
• whether to use subagents
• whether to use semantic code tools
• whether to force verification

Example:

yaml
routing:
  phases:
    exploration:
      model: gpt-5.4-mini
      reasoning: low
    risky-edit:
      model: gpt-5.5
      reasoning: medium
    test-debug:
      model: gpt-5.5
      reasoning: high

Long term, use eval results and CostLedger to learn the router.

11. Treat subagents as bounded workers with merge protocols

The agents tool is promising. To make it excellent for coding:

• Give subagents isolated worktrees or sandbox subdirectories by default.
• Require each subagent to return:
  • conclusion
  • evidence
  • changed files
  • verification
  • confidence
  • open risks
• Parent agent should get a structured result, not only final text.
• Add merge policy:
  • no direct modification by subagent unless requested
  • or subagent patch is staged for parent review
  • conflict detection

Good use cases:

• one subagent investigates tests
• one reads architecture
• one drafts implementation plan
• one reviews final diff

This can make the main agent much stronger on complex tasks.

12. Add a “reviewer agent” mode before final answer

For medium/large code changes, automatically run a cheap reviewer pass:

Reviewer checks:

• Did we satisfy the request?
• Are changes scoped?
• Did tests actually run?
• Any likely regressions?
• Any final answer overclaiming?
• Any missed docs/types/tests?

This can be a subagent or same model with a constrained prompt. The reviewer should inspect the diff and verification output.

This is one of the highest ROI quality features.

A bad context-pruning decision can silently degrade coding performance, so observability is key.

This turns “improve yourself” from a philosophical idea into an engineering system.
