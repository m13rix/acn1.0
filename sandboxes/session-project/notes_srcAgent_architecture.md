Для подготовки исследовательской основы я провел анализ кодовой базы проекта ACN1.0. Ниже представлен структурированный обзор актуальной архитектуры, основанный на анализе `src/core`, `tools/`, `src/sandbox` и конфигурационных файлов.

---

### 1. Provider Tools и эволюция вызова инструментов
**Подтверждено по коду:** В системе сосуществуют два подхода. Старый (Legacy) — жестко прописанные в коде функции. Новый (Provider-based) — описание инструментов через YAML-манифесты (`tools/<name>/tool.yaml`) и их динамическая загрузка.
*   **Механизм:** Инструменты теперь являются независимыми ESM-модулями. Переход к ним важен, так как это позволяет унифицировать интерфейс между разными LLM-провайдерами (OpenRouter, Gemini, Anthropic).
*   **Важность:** Это позволяет реализовать "Native Tool Calling" (когда модель сама генерирует JSON для вызова) вместо парсинга текста, что повышает надежность системы.
*   **Ограничение:** В `src/providers` видно, что не все провайдеры одинаково поддерживают стриминг и сложные вызовы (см. `config/no-stream-models.json`).

### 2. Принцип Code-as-Action
**Подтверждено по коду:** Это фундаментальный принцип системы, реализованный в `src/sandbox/LocalSandbox.ts`.
*   **Механизм:** Вместо того чтобы просто выдавать текст, агент генерирует блоки кода на TypeScript/JavaScript. Этот код исполняется в реальном времени через `tsx` в изолированной директории `sandboxes/session-<id>`.
*   **Реальные механизмы:** В `Executor.ts` виден цикл: `Prompt -> LLM -> Code Block -> Sandbox Execution -> Observation (stdout/stderr) -> Prompt`.
*   **Отличие:** Агент не ограничен набором кнопок; он может писать алгоритмы для обработки данных "на лету".

### 3. Оркестрация агентов и подагентов
**Подтверждено по коду:** Реализовано через инструмент `tools/agents/`.
*   **Механизм:** Главный агент (обычно TELOS из `agents/core/`) может порождать новые сессии других агентов (`researcher`, `analyzer`, `director`). 
*   **srcAgent:** Это специальный инструмент (`tools/srcAgent/`), который вызывает "архитектурного агента" (меня) для модификации самого исходного кода системы. Это создает петлю саморазвития (Self-Evolution).
*   **Вероятная интерпретация:** Система движется к иерархической структуре, где `director` распределяет задачи, а `researcher` собирает данные.

### 4. Heartbeat и Event-Driven Proactivity
**Подтверждено по коду:** Директория `src/heartbeat/` и `tools/heartbeat/`.
*   **Механизм:** Система имеет фоновый цикл ("сердцебиение"). Это позволяет агенту не просто отвечать на запросы, а инициировать действия по таймеру или событию.
*   **Реальные механизмы:** В `data/heartbeat/tasks.json` хранятся запланированные задачи. 
*   **Научно-техническая новизна:** Переход от реактивной модели (User -> Agent) к проактивной (System -> Agent), где агент обладает "чувством времени" и автономным циклом жизни.

### 5. Система памяти: Skills, Notepad и LanceDB
**Подтверждено по коду:**
*   **Skills (Долговременная):** Используется векторная база данных `LanceDB` (хранится в `data/skills/`). Это фрагменты кода или знаний, которые агент может "вспомнить" через семантический поиск.
*   **Notepad (Рабочая):** Инструмент `tools/notepad/` (`notes.json`). Используется для хранения текущих планов, промежуточных данных и заметок, которые не должны потеряться при смене контекста LLM.
*   **Global Memory:** В `data/memory/` видны файлы `global_memory_facts.lance`, что указывает на попытку создать общее пространство знаний между всеми агентами.

### 6. Sandbox Environments и работа с пакетами
**Подтверждено по коду:** `src/sandbox/LocalSandbox.ts`.
*   **Механизм:** Каждая сессия получает свою папку в `sandboxes/`.
*   **Работа с пакетами:** В коде песочницы предусмотрена возможность установки зависимостей через `npm install` прямо в процессе выполнения задачи, если агенту не хватает библиотек для вычислений.
*   **Ограничение:** На Windows (win32) запуск процессов через `tsx` может иметь задержки и проблемы с правами доступа, что видно по обработчикам ошибок в коде.

### 7. Обзор специфических систем
*   **Strategy System:** Реализована в `tools/strategy/`. Позволяет агенту сохранять сложные графы выполнения задач (`data/strategy/strat_*.json`).
*   **Realtime:** Попытка интеграции голосового/потокового взаимодействия (вероятно, через OpenAI/Gemini Realtime API).
*   **Homework:** Специализированный модуль (`src/homework-library/`) для обработки образовательного контента, включая OCR (скрипт `homework_pdf_ocr.py`).
*   **Files/View:** Утилиты для навигации по файловой системе без необходимости читать каждый файл целиком (экономия контекста).

### 8. Новизна vs Развитие
*   **Развитие:** Использование векторных БД (RAG) и Tool Calling — это стандарт индустрии.
*   **Новизна:** 
    1.  **Глубокая интеграция Code-as-Action:** Система не просто вызывает API, а манипулирует своим окружением через синтез кода.
    2.  **Саморедактирование через srcAgent:** Реализованный механизм, позволяющий агенту предлагать и вносить изменения в свой собственный `src/`.
    3.  **Архитектура "Heartbeat":** Эмуляция автономного биологического ритма для ИИ-системы.

### 9. Ограничения и незавершенные части
*   **Неясность:** В `data/memory/debug/` много пустых файлов, что указывает на стадию активной отладки системы логирования памяти.
*   **Ограничение:** Контекстное окно LLM все еще является "бутылочным горлышком", несмотря на наличие Notepad.
*   **Незавершенность:** Модуль `realtime` содержит много `TODO` и скелетных реализаций, что говорит о его экспериментальном статусе.
*   **Legacy Code:** В папке `old_examples/` хранится большой объем кода, который еще не полностью портирован на новую ESM-архитектуру провайдеров.

**Резюме для проекта:** Система ACN1.0 представляет собой гибрид классической агентной платформы и саморазвивающейся среды исполнения кода, где ключевым фактором эволюции является переход от текстовых ответов к автономному программированию действий.
