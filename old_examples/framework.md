# Концептуальная спецификация мультиагентного фреймворка

**Дата записи:** 2025-10-07

---

## 1. Введение — цель и мотивация

Проект представляет собой исследовательский мультиагентный фреймворк с иерархической организацией агентов, контейнеризованной общей средой и механизмами автоматического подбора оптимальных моделей для агентов нижнего уровня. Главная цель — создание системы, которая сможет решать **сложные, многослойные и жизненно важные задачи**, где важнее получить корректный результат, чем минимальная задержка.

Ключевые мотивации:

* Экспериментировать с иерархической (не плоской) организацией агентов для повышения качества координации и надёжности.
* Обеспечить адаптивность: автоматически подбирать наилучшие модели/инструменты для специфичных задач.
* Поддерживать единое виртуальное/контейнеризованное пространство данных и файлов для всех агентов.
* Сделать систему пригодной как для исследовательской работы (метрики, сравнения), так и для прикладного использования (персональный помощник, мониторинг жизни пользователя).

---

## 2. Высокоуровневые принципы дизайна

1. **Иерархичность вместо плоскости.** Агенты устроены в уровнях: *верхний оркестратор* → необязательные средние уровни → *последний слой элементарных агентов*. Первый и последний слои — обязательны.

2. **Результат важнее скорости.** Для критичных задач система предпочитает более тщательную проработку (больше вычислений, более мощные модели), даже если это займёт больше времени.

3. **Последовательное выполнение по умолчанию.** Задачи в верхних и внутренних слоях выполняются **последовательно** (аналогично проектированию и постройке здания: сначала план, потом реализация, потом проверка). Исключение — агенты последнего слоя (простые независимые задачи) — они могут выполняться **параллельно**.

4. **Модульность и масштабируемость.** Количество внутренних слоёв технически не ограничено, но по умолчанию система проектируется с 0–2 средними уровнями (чтобы не усложнять архитектуру).

5. **Разделение ролей в агенте (кроме последнего слоя).** Каждый агент (кроме элементарных) состоит из двух частей: **планировщик** и **исполнитель** (executor). Планировщик — мощная модель, формирует план / to‑do список; исполнитель — лёгкая модель, выполняет шаги.

6. **Простота синтаксиса для последнего слоя.** Формат команд/инструкций для агентов последнего уровня должен быть максимально компактным и простым (минимум токенов), чтобы его могли корректно понимать и самые лёгкие модели.

7. **Гибкость вызова последнего слоя.** Агенты последнего уровня может вызвать **любой** агент системы, не только предпоследний.

8. **Эксперименты с коммуникацией.** Возможность экспериментировать с общим контекстом/контейнером и прямыми коммуникациями между агентами последнего уровня (несколько агентов над одной задачей, общий публичный контекст и т.п.).

9. **«Неучитываемые» агенты внутри инструментов.** Некоторые инструменты могут включать свои вложенные автономные агенты (например, LLM внутри инструмента веб-поиска). Эти агенты не входят в основную иерархию и учитываются отдельно.

## 3. Структура слоёв и роли

### 3.1 Первый слой — Общий (root) агент

* Обязательный центральный агент — входная точка для задач, общий наблюдатель и координатор.
* Решает, какие задачи направлять во внутренние слои, какие — передавать в систему детекторов/фильтров, и когда привлекать человеческий контроль.

### 3.2 Внутренние (средние) слои — Оркестраторы

* (Опционально) 0..N промежуточных оркестраторов.
* Каждый оркестратор может быть разделён на планировщик + исполнитель (сильная/легкая модель).
* Отвечают за декомпозицию задач, агрегацию результатов от нижних слоёв и валидацию промежуточных результатов.

### 3.3 Последний слой — Элементарные агенты (primitive agents)

* Обязательный слой, выполняет простые и узкоспециализированные действия: расчёты, поиск в интернете, генерация изображений, ответы на вопросы, заполнение форм, и т. п.
* Может содержать **набор инструментов** (calc, Wolfram API, веб-поиск и т.д.).
* Агентов последнего слоя можно вызывать параллельно (по необходимости).
* Их интерфейс — **минималистичный**: простой синтаксис команд, минимальное количество токенов.
* Выбор **конкретной модели** и необходимых инструментов для каждого агентa последнего слоя осуществляется автоматической системой подбора моделей (см. раздел 6).

---

## 4. Внутреннее устройство агента (планировщик + исполнитель)

### 4.1 Планировщик

* Более мощная модель, получает задачу + контекст и формирует:

    * **Идеальный план** (стратегия достижения цели).
    * **To‑do список** — пошаговый набор задач/действий, которые должны быть выполнены исполнителем.
    * Рекомендации по инструментам и выбору моделей для шагов последнего слоя.
* План составляется один раз (или пересматривается при изменении контекста).

### 4.2 Исполнитель

* Лёгкая/оптимизированная модель, получает только необходимый контекст + текущий to‑do шаг.
* Обязан чётко следовать синтаксису и не «угадывать» лишнего.
* Выполняет шаги последовательно, обновляет состояние to‑do списка (отмечает завершённые шаги).

### 4.3 To‑do списки и Context Engineering

* To‑do список — основной механизм передачи задач от планировщика исполнителю.
* При завершении шагов система автоматически вычеркивает пункты и передаёт обновлённый список исполнителю через engineered context / системный промпт.
* Это снижает риск «забывания» шагов при большом объёме контекста и повышает детерминированность поведения.

---

## 5. Поток обработки входящих данных и детекция значимых изменений

### 5.1 Источники данных

* Постоянно обновляющаяся база данных: чаты (WhatsApp, Telegram), заметки, дневник, записи разговоров, файлы, логи и т.д.

### 5.2 Небольшая модель‑детектор (fast detector)

* Как только накопилось некоторое количество новых данных за день (порог настраиваемый), запускается облегчённая модель‑детектор.
* Детектор обучен на распознавании «важных изменений» / потенциально отклоняющихся событий (настроенная на конкретного пользователя или общая).

### 5.3 Предиктивная модель

* Если детектор отмечает что‑то значимое, информация передаётся в предиктивную модель (у пользователя уже есть такая модель), которая прогнозирует ожидаемое поведение и оценивает степень отклонения.

### 5.4 Главный агент (рубеж принятия решения)

* На основе комбинированной оценки (детектор + предиктивная модель) проводится взвешивание: стоит ли поднимать сигнал до главного оркестра (чтобы тот инициировал план действий / уведомил пользователя и т. п.).

---

## 6. Автоматическая система подбора моделей для агентов последнего уровня

### 6.1 Цель

* Для каждой конкретной задачи последнего слоя автоматически выбирать **лучшую модель** из доступных провайдеров (OpenRouter, OpenAI, локальные модели и т.д.) с учётом качества, стоимости и скорости.

### 6.2 Компоненты системы выбора

1. **Каталог/реестр доступных моделей** (с метаданными: возможности, среда, токены/латентность, цена, лицензия, ограничения).
2. **Скоринговая функция** — комбинированная метрика, учитывающая: релевантность (оценка качества), задержку, стоимость (цена вызова), потребление токенов, ограничение бюджета и историческую надёжность.
3. **Фильтры политик** — правила, например: "не использовать модели дороже X за задачу" или "для математических задач предпочитать Wolfram‑backed или специализированные вычислительные API".
4. **Быстрая LLM‑модель** — модель, натренированная выбирать подходящую модель + инструменты для конкретной задачи (она же может выдавать rationale и предложение по выбору).
5. **Кэш/мемоизация** — запоминать хорошие соответствия (задача → модель), чтобы не пересчитывать заново.

### 6.3 Учет финансовых ограничений

* В реальной эксплуатации нужно ввести **ограничения по стоимости**: бюджет‑cap, предельная цена за вызов, приоритет качества/цена.
* Скоринговая функция должна возвращать компромисс «качество / стоимость». Можно ввести параметр `cost_sensitivity`, который регулируется пользователем.

---

## 7. Инструменты (tools) и встроенные агенты в инструментах

* Агенты последнего уровня могут обладать набором инструментов: калькулятор, Wolfram API, веб‑поиск, PDF‑ридер, генератор изображений и т.д.
* Не все инструменты доступны всегда (экономия). Планировщик определяет оптимальный набор инструментов для задачи.
* Некоторые инструменты могут включать свои автономные LLM‑агенты (внутренние/неучитываемые агенты). Они выполняют работу «под капотом» и не входят в основную иерархию.

---

## 8. Общая виртуальная среда (контейнеризация) и менеджер контекста

### 8.1 Контейнеризация и единая рабочая среда

* Вся система работает в единой виртуальной/контейнеризованной среде (например, Docker / lightweight VMs), где у агентов есть общий доступ к файлам, артефактам, логам и результатам.
* Все создаваемые файлы (отчёты, презентации, изображения и др.) сохраняются в этой среде и доступны через специальный инструмент.

### 8.2 Менеджер контекста (Context Manager)

* Специальный инструмент, который позволяет агентам просматривать, использовать и обновлять документы других агентов.
* Может реализовывать версии (versioning), доступ/разрешения и индексацию содержимого для быстрой выборки.
* Общий контекст может быть общим для всех агентов и/или для всех сессий (настраиваемо).

---

## 9. Связь между агентами и экспериментальные режимы коммуникации

* По умолчанию основная коммуникация проходит по иерархическому каналу (выше → ниже → выше).
* Экспериментально можно включать прямые коммуникации между агентами последнего слоя и/или общий публичный контекст, куда они могут «скидывать» промежуточные результаты.
* Возможности эксперимента:

    * Несколько агентов решают одну задачу (ensemble / diverse approaches).
    * Разные агенты работают над частями большой задачи и публикуют результаты в общем контексте.
* Эти режимы могут быть ресурсозатратны и дорогие (много параллельных вызовов, больше токенов).

---

## 10. Формат сообщений / токенов и подход React (Thought / Action / Observation)

* В качестве стандартного синтаксиса коммуникации предлагается использовать аналог React‑подхода: **Thought**, **Action**, **Observation** (или переименовать токены по вкусу).
* Это понятный и выверенный формат: планировщик формулирует Thought → выдаёт Action (конкретные шаги / запуск инструментов) → исполнитель/инструмент возвращает Observation.
* Для агентов последнего уровня синтаксис Action сокращается до минимума (минималистичный формат команд).

---

## 11. Безопасность, конфиденциальность и этика

Особенно важно при использовании личных данных (чаты, дневники, звонки):

1. **Шифрование и хранение.** Все личные данные должны храниться зашифрованными, с доступом только для доверенных компонентов.
2. **Гранулярный контроль доступа.** Возможность включать/отключать источники данных (например, не индексировать голосовые записи без явного согласия).
3. **Анонимизация для исследований.** Для представления результатов в исследовательской работе — анонимизировать или синтезировать примеры; не включать реальных чувствительных подробностей без разрешения.
4. **Журнал действий (audit trail).** Логировать все действия агентов, кто что сделал, какие решения были приняты.
5. **Ограничения по автономным действиям.** Для действий, которые могут иметь реальные последствия (финансы, отправка сообщений от имени пользователя), ввести «человеческий‑подписант» или строгие правила автозапуска.
6. **Этическая проверка.** При использовании системы в реальных сценариях — проходить проверку на безопасность/побочные эффекты.

## 12. Метрики, эксперименты и оценка (для научной работы)

### 12.1 Основные метрики

* **Качество ответа / точность** (task success rate).
* **Время до решения** (latency), но как вторичный показатель.
* **Стоимость** (USD / вычислительные ресурсы) на задачу.
* **Надёжность / устойчивость к сбоям.**
* **Частота ложных срабатываний (детектор)** — FP / FN.
* **Пользовательская удовлетворённость** (при интерактивных сценариях).

### 12.2 Экспериментальные сценарии

1. **Иерархическая vs плоская архитектура.** Сравнить качество и стоимость на наборе задач (сложные планировочные задачи, жизненно важные сценарии).
2. **Планировщик+Исполнитель vs монолитный агент.** Оценить преимущества по стабильности и воспроизводимости.
3. **Автоматический подбор модели vs фиксированная модель.** Измерить улучшение качества и влияние на стоимость.
4. **Влияние общего контекста и меж‑агентного общения.** Посмотреть, влияет ли общий контекст на скорость/качество и при каких условиях экономически целесообразен.
5. **Чувствительность к бюджету.** Проверить различные `cost_sensitivity` настройки.

### 12.3 Данные для тестирования

* Синтетические задачи и датасеты.
* Анонимизированные личные логи (в случае соглаcия) или смоделированные сценарии.
* Стандартные наборы задач (например, math problems, information retrieval, planning tasks).

## 13. Практические заметки по реализации (инструменты и подходы)

* Контейнеризация: Docker для изоляции агентов / общей среды; опционально Kubernetes для масштабирования экспериментов.
* Хранилище данных: сочетание базы данных (Postgres/Mongo) + объектного хранилища (S3‑совместимое) для артефактов.
* Индексация контента: векторная база (например, Milvus, Pinecone, Weaviate) для быстрого поиска по документам/контексту.
* Менеджер контекста: сервис с API, управляющий доступом, версиями и выдачей подходящего контекста для моделей.
* Система подбора моделей: реестр доступных моделей + scorer сервис (проводит сравнительный анализ и принимает решение).
* Лёгкие локальные модели для детекции/быстрой классификации: Tiny/Distilled LLMs (on‑device) или small transformer / distilled checkpoints.
* Логирование и наблюдаемость: ELK/Prometheus + dashboard для отслеживания решений агентов и затрат.

---

## 14. Управление затратами (важно)

Рекомендации для контроля расходов:

* Ввести глобальный бюджет и ежедневный лимит на вызовы платных моделей.
* Ограничения на максимальную цену за один вызов.
* Применять приоритеты: сперва дешёвые/локальные модели, потом платные, если качество критично.
* Кеширование результатов дорогих вычислений.
* Метрики ROI для каждого класса задач — чтобы обоснованно разрешать дорогие вызовы.

---

## 15. Рекомендации для оформления исследовательской работы и этических аспектов

1. Для школьного/олимпиадного проекта (9 класс) фокусируйся на **архитектуре, методах и метриках** (а не на демонстрации реальных личных данных). Это повысит воспроизводимость и упростит рецензирование.
2. Реальные жизненные кейсы можно демонстрировать отдельно в виде анонимных/синтетических сценарииев или приватной демонстрации на защите (с учётом ГОСТ/политик школы/конкурса).
3. Включи в работу следующие разделы: постановка задачи, архитектура, экспериментальная методология, результаты и обсуждение, ограничения и дальнейшие направления.
4. Для грантов: подготовь краткое описание (one‑pager) с чёткими целями, вкладом и планом работ; укажи образовательную и общественную ценность проекта.

## 16. Дорожная карта (минимальный MVP → расширение)

**MVP (школьный проект):**

* Реализовать базовую иерархию: root agent → 1 внутренний оркестратор → набор элементов последнего слоя.
* Интеграция контейнеризированной общей среды с менеджером контекста.
* Простая система выбора моделей: реестр + скорер с учётом цены и релевантности (минимальная версия).
* Базовые эксперименты: сравнение иерархии vs плоской структуры на 2–3 задачах.

**Следующие шаги:**

* Улучшение детектора/предиктивной модели, более точная кастомизация под пользователя.
* Добавление экспериментов с общим контекстом и параллельной работой агентов последнего слоя.
* Введение гибких политик бюджетирования и подробного логирования.

---

## 17. Резюме — ключевые решения, зафиксированные в документе

* Использовать иерархическую мультиагентную архитектуру с обязательными первым и последним слоями.
* Разделять агента (кроме последнего слоя) на планировщик + исполнитель.
* Применять to‑do списки, передаваемые исполнителям через Context Engineering.
* Большинство слоёв работает последовательно; последний слой — параллельно.
* Контейнеризированная общая среда с менеджером контекста для совместного хранения артефактов.
* Автоматическая система выбора лучшей модели для агентов последнего слоя (с учётом стоимости).
* Возможность экспериментировать с рест‑коммуникацией между агентами и душой общего контекста.
* Встроенные в инструменты автономные «неучитываемые» агенты допускаются.
* Фокус исследовательской работы — архитектура и метрики; реальные личные данные анонимизировать.

## 18. Приложение: подробный список зафиксированных пунктов (быстрая сводка)

1. React‑подход: Thought / Action / Observation (можно переименовать).
2. Постоянная база данных: чаты, заметки, дневник, записи разговоров.
3. При достижении порога новых данных запускается маленькая модель‑детектор.
4. Детектор передаёт сигнал предиктивной модели, та измеряет отклонение от предсказания.
5. Взвешивание результатов детектора и предиктивной модели → решение передать задачу главному агенту.
6. Система автоматически подбирает лучшие модели для агентов последнего слоя (OpenRouter и др.).
7. У системы есть механизмы ограничения затрат (бюджет, max price, cost_sensitivity).
8. Агенты последнего уровня — полноценные агенты с инструментами; инструменты выбирает быстрый LLM.
9. Синтаксис для агентов последнего уровня — максимально прост.
10. Любой агент может вызвать агента последнего уровня.
11. Все агенты (кроме последнего уровня) имеют планировщик и исполнитель.
12. Планировщик создаёт to‑do списки; исполнитель получает только оставшийся список шагов.
13. Общая контейнеризированная среда и менеджер контекста — единое хранилище артефактов.
14. Возможность экспериментировать с коммуникацией между агентами; это может быть дорого.
15. «Неучитываемые» агенты — автономные агенты внутри инструментов.
16. Синхронное/последовательное выполнение на уровнях выше последнего; параллелизм только у последнего.
